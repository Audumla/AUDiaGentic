# PKT-RLS-009 — Incremental release/ledger updates for new contracts

**Phase:** Phase 2.1  
**Primary owner group:** Release and Ledger

## Goal

Extend release/ledger outputs to reflect new contract fields introduced by later phases while preserving Phase 2 determinism.

## Why this packet exists now

As new contract fields appear (e.g., provider model selection metadata), release summaries and audits may need to include or ignore them explicitly. This packet provides a controlled update point without changing Phase 2 core behavior.

## Dependencies

- Phase 2 gate `VERIFIED`
- `PKT-FND-008` for schema updates if release artifacts require new fields

## Concrete release inputs

This packet must account for new release-visible data points introduced by later phases, including:

- provider selection metadata from job/change events
- model identifiers or aliases when they are surfaced in audit text
- config-driven provider details when they appear in release summaries

## Ownership boundary

This packet owns the following implementation surface:

- `src/audiagentic/release/*`
- release tests under `tests/unit/release/`, `tests/integration/release/`, `tests/e2e/release/`
- tracked release docs generated by release scripts

### It may read from
- updated contracts in `docs/specifications/architecture/`
- current ledger fixtures

### It must not edit directly
- lifecycle, jobs, provider adapters, or overlay code

## Detailed build steps

1. Identify which new fields are release-visible versus internal-only.
2. Update release summary and audit generation to include or explicitly ignore those fields.
3. Keep ledger sync deterministic after the new fields are present in source data.
4. Ensure exactly-once finalization still ignores fields that do not belong in the ledger.
5. Add tests that cover the new fields in release outputs.

## Integration points

- `src/audiagentic/release/fragments.py`
- `src/audiagentic/release/sync.py`
- `src/audiagentic/release/current_summary.py`
- `src/audiagentic/release/audit.py`
- `src/audiagentic/release/finalize.py`
- `tests/integration/release/`

## Tests to add or update

- release summary tests
- audit summary tests
- end-to-end release flow tests if outputs change

## Acceptance criteria

- release outputs remain deterministic with new fields present
- ledger sync and finalization behavior unchanged
- tests prove new fields are handled correctly
- release summaries make it explicit when later-phase fields are omitted or summarized

## Recovery procedure

If this packet fails mid-implementation:
- revert release code changes
- rerun `python -m pytest tests/unit/release tests/integration/release tests/e2e/release`

## Parallelization note

This packet may run in parallel only with work that does not touch release modules or tracked release docs.

## Out of scope

- schema changes (owned by `PKT-FND-008`)
- provider selection or job logic changes
