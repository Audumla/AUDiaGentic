# Standard

Default standard for integrating a new provider (agent, LLM, or tool surface) into AUDiaGentic-based projects. Covers rule file structure, output format determination, generator logic, synchronization policy, and verification.

# Source Basis

This standard is derived from the repository's existing provider integration pattern and rule generation tooling.

Sources:

- Repository rule generator: `tools/config/generate_provider_rules.py`
- Repository rule system: `.audiagentic/config/rules/shared/` and `.audiagentic/config/rules/providers/`
- AgentRunner pattern: `H:\development\projects\AgentRunner\.agent_runner\config\rules\`
- Related: standard-0001 (planning documentation), standard-0008 (Python implementation)

# Requirements

1. Each provider integration must create provider-specific rule files under `.audiagentic/config/rules/providers/{provider-name}/`:

   - `01-provider-specific.md` — provider constraints, capabilities, authentication notes, rate limits
   - Additional rule files as needed, numbered sequentially

2. The output file format and path must be determined and documented before implementation:

   | Provider     | Output File                       | Format          |
   | ------------ | --------------------------------- | --------------- |
   | Claude       | `CLAUDE.md`                       | Markdown        |
   | Codex        | `CODEX.md`                        | Markdown        |
   | Cline        | `.clinerules`                     | Cline-specific  |
   | Copilot      | `.github/copilot-instructions.md` | Markdown        |
   | New provider | Defined in spec                   | Defined in spec |

3. Generator logic must be implemented or extended in `tools/config/generate_provider_rules.py`:

   - Add a `generate_{provider_name}_rules(shared_rules, provider_rules)` function
   - Register the provider in the `PROVIDERS` dict with its output file path
   - Generated output must include a header comment identifying it as generated by AUDiaGentic

4. The new provider must be added to the synchronization list in `CLAUDE.md` so regeneration is documented and discoverable.

5. Provider rule files must not contradict shared rules. Conflicts between shared and provider-specific rules must be resolved at authoring time, not at generation time.

6. Provider integrations must comply with standard-0011 (component architecture) for any new generator modules: the generator function is a Surface-layer concern and must not contain business logic beyond rule merging and output formatting.

7. Provider-specific behavior notes must be documented in `.audiagentic/config/PROVIDERS.md`:

   - output file path
   - format
   - generation command
   - special considerations or known limitations

# Default Rules

- Keep provider-specific rules minimal; prefer shared rules where behavior is common across providers.
- Generated files must always include a header comment marking them as generated so projects do not manually edit them.
- Run the generator after any shared rule change and commit both the rule change and the regenerated output.
- Do not hand-edit generated provider output files; edit the source rules instead.

# Verification Expectations

- Run `python tools/config/generate_provider_rules.py` and confirm it exits cleanly.
- Confirm the output file exists at the expected path.
- Confirm the output includes shared rules and provider-specific rules without conflicts.
- Confirm the output header comment is present.
- Confirm `CLAUDE.md` lists the new output file in its sync policy.
- Test that the provider can read and apply the rules in practice (no contradictions or parse failures).

# Completion Checklist

- [ ] Provider-specific rule files created under `.audiagentic/config/rules/providers/{provider}/`
- [ ] Output file format and path defined
- [ ] Generator function implemented and registered in `PROVIDERS` dict
- [ ] Generator runs cleanly and produces output at expected path
- [ ] `CLAUDE.md` updated to list new output file
- [ ] Generated file has proper header comment
- [ ] Provider documented in `.audiagentic/config/PROVIDERS.md`
- [ ] Shared rules merged correctly with no conflicts
- [ ] Tested with actual provider

# Non-Goals

- Defining provider-specific agent behavior beyond rule delivery.
- Replacing project-specific provider configuration or credentials management.
- Automating provider testing or integration validation beyond rule application.
