# Description

Implement replay support for persisted events. This task owns reading stored event envelopes, filtering when requested, re-publishing in chronological order, and marking replayed deliveries so consumers remain idempotent.

**API:**
```python
replay_service.replay(
    from_timestamp="2026-04-14T00:00:00Z",
    to_timestamp="2026-04-14T23:59:59Z",
    event_type_pattern="planning.*"
)
```

**Key behaviors:**
- Chronological order: events re-published in timestamp order
- `is_replay: true`: replayed events marked in envelope
- **Opt-in dispatch:** Subscribers receive replayed events only if `config.replay.dispatch_on_replay: true` (default: false)
- No re-persist: replay doesn't create duplicate files
- Filtering: support timestamp range and event type pattern

# Acceptance Criteria

- `replay()` reads persisted events and re-publishes via EventBus in chronological order
- Replayed events have `is_replay: true` in envelope
- Replay supports filtering by timestamp range and event type pattern
- Replay doesn't re-persist events (no infinite loops)
- **Opt-in replay:** Subscribers receive replayed events only if `config.replay.dispatch_on_replay: true`
- Malformed event files handled gracefully (logged, skipped)
- Unit tests cover: chronological replay, `is_replay` flag, filtering, no-repersist, opt-in behavior
- Smoke test proves replay imports and executes on sample persisted events

# Notes

Consumers must check `is_replay` flag for idempotency (task-0251, task-0261).
