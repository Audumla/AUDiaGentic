# Description

When planning items are deleted (soft or hard), the persisted counter must be synchronized to ensure the next ID generation is correct. This does NOT mean decrementing the counter to prevent gaps - IDs must remain stable and monotonic for auditability.

Instead, this task implements automatic calls to `sync_counter()` after deletions to ensure the counter reflects the highest existing ID, even if gaps exist.

# Acceptance Criteria

1. After any delete operation (soft or hard), `sync_counter()` is called automatically
2. Counter reflects highest existing ID (gaps are allowed and preserved)
3. Next ID generation after delete produces correct sequential ID
4. No decrement logic that would reuse deleted IDs
5. Works for all planning kinds (request, spec, plan, task, wp, standard)

# Notes

- **Do not** implement "decrement to prevent gaps" - this breaks auditability
- **Do** implement "sync to highest existing ID" - this ensures correctness
- Gaps in IDs are acceptable and expected when items are deleted
- The counter file stores the highest ID seen, not the count of items
