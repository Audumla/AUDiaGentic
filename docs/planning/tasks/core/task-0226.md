# Description

Define and implement consistent stale-index handling in `lookup(id_)` and `head(id_)` for two failure cases:

**Index file missing** — `lookup.json` does not exist yet (e.g. `index()` has never been run). Fall back to a full `scan_items()` pass to locate the item. `lookup(id_)` should return the parsed item, and `head(id_)` should return the equivalent lean metadata derived from the located item. Log a warning that the index is missing.

**Id not found in index** — the item exists on disk but the index is stale. Fall back to `scan_items()`, locate the item, and return it. `lookup(id_)` should return the parsed item, and `head(id_)` should return the equivalent lean metadata derived from the located item. If still not found after the fallback scan, raise a clear `ValueError(f"{id_} not found")`.

Do not silently repair the index during a lookup — callers who need a fresh index should run `index()` or `reconcile()` explicitly. Document this in the method docstrings.

# Acceptance Criteria

- `lookup(id_)` succeeds when `lookup.json` is missing by falling back to scan
- `head(id_)` succeeds when `lookup.json` is missing by falling back to scan and returning lean metadata
- `lookup(id_)` succeeds when an id is absent from a stale index by falling back to scan
- `head(id_)` succeeds when an id is absent from a stale index by falling back to scan and returning lean metadata
- `lookup(id_)` raises `ValueError` with a clear message when the item does not exist at all
- `head(id_)` raises a clear error when the id is not in the index and does not exist on disk
- Fallback behaviour is consistent between `lookup()` and `head()`
- Tests cover missing index, stale index (id absent), and genuinely missing item cases

# Notes

Depends on task-0222. The fallback to scan preserves correctness at the cost of performance — callers who care about performance should ensure the index is current before calling.
