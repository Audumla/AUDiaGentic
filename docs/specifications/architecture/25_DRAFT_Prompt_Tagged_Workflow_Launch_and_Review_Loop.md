# DRAFT - Prompt-Tagged Workflow Launch and Review Loop

## Purpose

AUDiaGentic should be able to launch workflow activities from a tagged prompt in either CLI or VS Code, and it should support a review loop where one agent reviews another agent's work and returns structured feedback.

This is a Phase 3.2 extension. It follows the job core and workflow profile contracts already established in Phase 3.

## Why this exists

The user goal is to:
- write one prompt to plan work
- write another prompt to execute it
- write a review prompt to validate completeness and correctness

That requires a tagged prompt entry point rather than relying only on direct job creation.

## Scope

- prompt tag syntax for workflow activity selection
- prompt-to-job launch contract
- prompt source provenance for CLI and VS Code
- review-stage feedback loop
- stage handoff between `plan`, `implement`, and `review`

## Out of scope

- natural language intent parsing without tags
- automatic cross-agent delegation
- multi-agent fan-out
- provider model selection rules

## Prompt launch contract (draft)

```json
{
  "contract-version": "v1",
  "prompt-id": "prm_20260330_0001",
  "source": {
    "kind": "interactive-prompt",
    "surface": "vscode",
    "provider-id": "claude",
    "session-id": "sess_001"
  },
  "tag": "plan",
  "target": {
    "kind": "packet",
    "packet-id": "PKT-JOB-007"
  },
  "workflow-profile": "standard",
  "job-id": null
}
```

Rules:
- `tag` selects the workflow activity.
- `source.surface` must preserve whether the launch came from CLI or VS Code.
- if `job-id` is null, the prompt creates a new launch.
- if `job-id` is present, the prompt resumes or advances an existing job.

## Allowed tags

Recommended initial tag set:
- `plan`
- `implement`
- `review`
- `audit`
- `check-in-prep`

Tag behavior:
- `plan` creates or updates a plan artifact
- `implement` executes the selected packet or stage
- `review` inspects another agent's output and produces findings
- `audit` checks release or policy compliance
- `check-in-prep` prepares release-facing summaries

## Review loop contract (draft)

Review stages should consume:
- the original packet or job record
- the prior stage output or artifact path
- the reviewer prompt source
- the current acceptance criteria

Review output should contain:
- pass/fail recommendation
- missing items
- risky areas
- suggested fixes or follow-up prompts

## Review loop behavior

The review loop should support these scenarios:
1. plan in one prompt, implement in another
2. implement in one prompt, review in another
3. review can be repeated after fixes until acceptance

The review step must remain deterministic enough to test against fixtures even when the content being reviewed is generated by another agent.

## Open questions

- Should the tag syntax be free-form text, a prefix token, or a fenced metadata block?
- Should tags launch jobs directly or generate a job request envelope for confirmation?
- Should review output be an approval, a rejection, or a structured feedback report by default?
- Should VS Code and CLI share the same launcher path or separate adapters?

## Phase 3.2 exit gate

- tag syntax is frozen
- prompt-to-job launch is deterministic
- review stage produces structured feedback
- CLI and VS Code preserve source provenance
