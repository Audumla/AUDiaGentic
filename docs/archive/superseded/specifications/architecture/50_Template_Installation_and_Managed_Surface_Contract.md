# 50 Template Installation and Managed Surface Contract

## Purpose

This specification defines the ownership and synchronization contract between AUDiaGentic as
the template source and a target project that installs or refreshes AUDiaGentic-managed
assets.

The goal is simple: a target-project developer must be able to tell which files are managed,
which files are seeded once and then owned locally, and which runtime artifacts should never
be treated as template content.

## Scope

This contract applies to:

- installable baseline assets copied or refreshed by lifecycle/bootstrap flows
- generated provider instruction/function surfaces emitted by the shared regeneration facade
- runtime-only artifacts written under `.audiagentic/runtime/`

It does not make provider execution outputs into tracked template content.

## Sync modes

### `required-managed`

AUDiaGentic owns the file. Sync or regeneration may overwrite it on the next run.

Examples:

- `.claude/skills/`
- `.agents/skills/`
- `.clinerules/skills/`
- `.gemini/commands/`
- `AGENTS.md`
- `CLAUDE.md`
- `GEMINI.md`
- `.clinerules/prompt-tags.md`
- `.claude/rules/prompt-tags.md`

### `create-if-missing`

AUDiaGentic seeds the file if absent. Once present, the target project owns local edits
unless another contract explicitly says otherwise.

Examples:

- `.audiagentic/project.yaml`
- `.audiagentic/prompt-syntax.yaml`
- `.audiagentic/providers.yaml`

### `generated-managed`

The file is generated by AUDiaGentic and may be refreshed automatically, but it is not a
hand-maintained template source.

Examples:

- `docs/releases/CURRENT_RELEASE.md`
- `docs/releases/AUDIT_SUMMARY.md`
- `docs/releases/CHECKIN.md`

### `runtime-only`

The file or directory is runtime state. It must not be committed as installable template
baseline content.

Examples:

- `.audiagentic/runtime/`
- `.audiagentic/runtime/jobs/`
- `.audiagentic/runtime/provider-catalogs/`

## Managed provider-surface rule

Provider instruction/function surfaces are generated outputs of the shared regeneration facade.
Their behavioral source stays generic under `.audiagentic/skills/`, while provider-specific
formatting stays with provider-owned renderer definitions.

Important boundary:
- `.audiagentic/skills/` is a canonical provider-function source, not a shared provider-ready
  skill format
- generated outputs under `.agents/skills/`, `.claude/skills/`, `.clinerules/skills/`,
  `.gemini/commands/`, `.opencode/skills/`, and future provider roots are separate
  provider-owned surfaces
- providers must not share one hand-authored generic surface file that risks semantic drift or
  path/format conflicts across providers

Direct edits to generated provider-surface outputs are non-canonical and may be overwritten on
the next regeneration run.

## Managed headers

All `required-managed` files must carry a top-of-file managed header.

| File type | Comment style | Example |
|---|---|---|
| Markdown (`.md`) | HTML comment | `<!-- MANAGED_BY_AUDIAGENTIC: do not edit directly. Re-run regenerate_tag_surfaces.py to update. -->` |
| Python / YAML | hash comment | `# MANAGED_BY_AUDIAGENTIC: do not edit directly. Re-run regenerate_tag_surfaces.py to update.` |

The managed header marks the file as an AUDiaGentic-owned output rather than target-project
authored content.

## File ownership guidance

### Target project owns directly

- project code and business logic
- project-specific configuration intentionally classified as `create-if-missing`
- local custom files outside the managed-output list

### AUDiaGentic owns directly

- shared generated provider-function surfaces
- shared tracked instruction surfaces
- template-controlled baseline assets explicitly classified as `required-managed`

## Overrides and local customization

If a target project needs behavior that conflicts with a `required-managed` output, the
override must be requested by changing the canonical source or the provider-owned renderer
definition, not by hand-editing the generated file in place.

If a project needs a local variation in a seeded config file, that file should live under a
`create-if-missing` rule rather than being converted into a silent managed overwrite target.

## Validation

`tools/check_baseline_assets.py --check-managed-headers` must verify that all
`required-managed` outputs carry the expected header style.

Lifecycle/bootstrap checks must also continue to enforce that `.audiagentic/runtime/` remains
outside the installable baseline set.
