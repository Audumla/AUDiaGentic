# PKT-RLS-010 — Release/audit handling for prompt and review metadata

**Phase:** Phase 2.2  
**Primary owner group:** Release and Ledger

## Goal

Define and implement deterministic release/audit handling for prompt provenance and structured review outcomes introduced by the .2 extension.

## Why this packet exists now

Prompt launch and review introduce useful runtime metadata, but that metadata must not leak into tracked release docs accidentally. This packet makes omission and summarization rules explicit before jobs start emitting the new fields.

## Dependencies

- Phase 2 gate `VERIFIED`
- `PKT-FND-009`

## Concrete release inputs

This packet must handle:

- prompt provenance on change/job events
- review report and review bundle outcome metadata
- check-in-prep stage artifacts that intentionally summarize review outcomes

## Ownership boundary

This packet owns the following implementation surface:

- `src/audiagentic/runtime/release/*`
- release tests under `tests/unit/release/`, `tests/integration/release/`, `tests/e2e/release/`
- tracked release docs generated by release scripts

### It may read from
- prompt/review contracts from `PKT-FND-009`
- jobs runtime artifact conventions from `PKT-JOB-008` / `PKT-JOB-009` docs

### It must not edit directly
- lifecycle modules
- prompt parser/launch modules
- provider adapters

## Detailed build steps

1. Classify prompt/review fields as internal-only or summary-eligible.
2. Ensure raw prompt text is never written into tracked release docs by default.
3. Ensure raw review report content is never written into tracked release docs by default.
4. Allow deterministic summarized review outcome handling when a check-in artifact requests it.
5. Add tests proving release output remains deterministic whether prompt/review fields are present or absent.

## Tests to add or update

- release summary tests for omission of raw prompt provenance
- audit summary tests for summarized review outcome handling
- end-to-end release flow tests if tracked output changes

## Acceptance criteria

- release outputs remain deterministic with prompt/review metadata present
- raw prompt text and raw review bundles are omitted by default
- summarized review outcomes are explicit when surfaced intentionally
- existing release core ownership boundaries remain intact

## Recovery procedure

If this packet fails mid-implementation:
- revert release code changes
- rerun `python -m pytest tests/unit/release tests/integration/release tests/e2e/release`

## Parallelization note

This packet may run in parallel only with work that does not touch release modules or tracked release docs.

## Out of scope

- changing job execution semantics
- changing contract schemas
- auto-generating commit text from review reports
